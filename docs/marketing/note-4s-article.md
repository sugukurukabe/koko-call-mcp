# 小さなプロトコルが、なぜ新しい価値を生むのか — MCP を、経済学者の「無形資産の4S」で読む

> 想定検索クエリ: 「MCP とは」「Model Context Protocol わかりやすく」「無形資産 4S」「Capitalism without Capital」「AI エージェント 経済」「MCP ビジネス価値」

## はじめに — 一行で言うと

本稿の主張は、一行で書けます。**Model Context Protocol（MCP）は、経済学者がもう 30 年近く語ってきた「無形資産（intangible assets）の経済特性」と、ほとんど同じ形をしている。** だからこそ、これは派手な機能の話ではなく、**価値の生まれ方の話** として読まれるべきです。

机のうえに置かれた一枚の請求書は、紙としてはほとんど価値を持ちません。インクと薄い紙でできた、数十円の物体です。

しかし、その紙が運んでいる「いつ・誰が・何を・いくらで」という情報は、会社の意思決定を変えます。資金繰りが変わり、来月の給与日が変わり、ときに採用の可否すら変わります。

紙そのものは安く、紙のうえに乗っている情報は重い。これは経済学者がかなり前から指摘してきた現象です。Jonathan Haskel と Stian Westlake は、この現象を『[Capitalism without Capital: The Rise of the Intangible Economy](https://press.princeton.edu/books/hardcover/9780691175034/capitalism-without-capital)』（Princeton University Press, 2017）で、**無形資産の4S** として整理しました。

派手な新機能の話をしません。**小さなプロトコルが、無形資産の経済特性とほぼ同じ形をしているという観察の話** をします。それが、なぜ「価値の生まれ方」を変えるのか、という話に着地します。

---

## 無形資産の4Sとは — 30 年近く語られてきた、4 つの S

無形資産の4Sは、ソフトウェア・ブランド・R&D・組織知のような無形資産（intangible assets）が、機械や建物のような有形資産と異なる挙動を取る理由を、四つの頭文字 S で整理したフレームワークです。出典は前述『Capitalism without Capital』（2017）。

| S | 名前 | ひとことで言うと |
|---|---|---|
| **Scalability** | スケーラビリティ | 一度作れば、追加コストほぼゼロで何度でも使える |
| **Sunkenness** | サンクネス | 一度投じた費用は、企業が消えれば回収しにくい（沈む） |
| **Spillovers** | スピルオーバー | 価値が作った人の外側に漏れ出して、他者にも流れる |
| **Synergies** | シナジー | 他の無形資産と掛け合わせると、価値が跳ねる |

経済学の文脈では、これら 4S は「測りにくく、課税しにくく、けれども現代経済の中心になりつつある」と語られてきました。OECD の Productivity Statistics（2024 年版）でも、先進国の企業投資のうち無形資産投資が有形資産投資を上回る年が増えたことが報告されています。

ここから先は、この 4S を **MCP に当てはめる地図** として使っていきます。

## Model Context Protocol（MCP）とは何か — 経営者向けに、ごく短く

Model Context Protocol（MCP）は、AI（Claude、Cursor、ChatGPT など）に、外部のデータと操作を渡すための「共通の差込口」を定めたオープンな仕様です。Anthropic が 2024 年 11 月に公開し、現在は Anthropic / OpenAI / Google / Microsoft のエコシステムで採用が広がっています。

USB の差込口を想像してください。USB が登場するまで、マウスはマウス専用、プリンターはプリンター専用、キーボードはキーボード専用のケーブルが必要でした。USB が出たあと、形が一つに揃ったので、機器メーカーも PC メーカーも、それぞれの工夫を「同じ形」のまま接続できるようになりました。

MCP は、AI に対する USB のようなものです。

- 会計のデータを AI に渡したい人は、会計の MCP サーバーを書きます
- 入札情報を AI に渡したい人は、入札の MCP サーバーを書きます
- 自社の社内データを渡したい人は、社内 MCP サーバーを書きます

書いた人は、それを「**同じ形**」で公開します。すると、Claude も Cursor も ChatGPT も、これから出てくる新しい AI も、**同じ差込口を通じて** 同じデータにアクセスできるようになります。

要点を一行で書くと、**MCP は「機能」を増やす規約ではなく、「並べ方」を揃える規約** です。この「並べ方を揃える」という地味な決定が、4S をすべて発火させる、というのが本稿の以降の話です。

## S1. Scalability — 一度公開すれば、文脈は何度でも呼ばれる

MCP の Scalability は、**ひとつのサーバーが、複数のホスト（Claude / Cursor / ChatGPT）から同じ形のまま呼ばれる**、という性質です。書く側にとってはクライアントごとの移植が要らず、使う側にとっては「どのアプリからでも、同じ業務文脈にアクセスできる」体験になります。

スケーラビリティ（Scalability）は、無形資産の最も語られる特徴です。ソフトウェアは 1 人に売っても 1 万人に売っても、追加生産コストがほぼ変わりません。同じことが、MCP サーバーで起きはじめています。

具体例を挙げます。

- [GitHub MCP](https://github.com/github/github-mcp-server) は、Issue や Pull Request を AI に渡します。同じサーバーが Claude Desktop でも Cursor でも、同じように動きます。
- [Stripe MCP](https://docs.stripe.com/mcp) は、決済データへの読み取りを AI に渡します。財務担当者が見るときも、エンジニアがデバッグするときも、呼ばれるツールは同じです。
- [Slack MCP](https://github.com/modelcontextprotocol/servers/tree/main/src/slack) は、組織のチャットに対する文脈を AI に渡します。

ここで起きているのは「API をラップしたツールができた」ではありません。**「文脈そのもの」がスケーラブルな単位になった**、ということです。

### 未来に起きそうなこと

会社ごとの就業規則、契約雛形、与信判断の基準、稟議の慣習。こうした「文書化されているけれど、社外からは触れない無形資産」が、MCP サーバーとして社内・取引先・関係省庁の間で共有されるようになる可能性があります。

そのとき、**会社の知識は、SaaS の画面ではなく、エージェント越しに反復利用される無形資産になります**。これは経営から見ると、ナレッジマネジメントの形を変える話です。

## S2. Sunkenness — MCP の価値は、外から見えない実装に沈む

MCP の Sunkenness は、ツール名と短い説明文の背後に、**ドメイン知識・出典管理・エラー設計・権限境界が「沈む」**、という性質です。READMEや UI の表面では見えないこれらの実装が、結局はエージェントの判断の質を決めます。

Haskel と Westlake は、無形投資の費用は「沈みやすい（sunk）」と表現しました。R&D や組織開発に投じたお金は、会社が傾いたとき、機械や建物のように転売できません。MCP サーバーにも、同じ性質があります。

外から見えるのは、ツール名と短い説明文だけです。たとえば `extract_bid_requirements` という名前と「PDF 仕様書から要件を抽出します」という一文。

しかし、その背後に沈んでいるものは小さくありません。

- 仕様書 PDF のテキスト抽出と、章構造の正規化
- 「参加資格」「評価基準」など、現場で使われる語の同義語処理
- 出典（ページ番号・原文）を必ず添える設計
- 失敗したときに、人間にとって意味のあるエラーメッセージを返す処理

これらは、UI にも README にもほとんど表に出ません。けれども、これがあるかどうかで、エージェントの判断の質は決定的に変わります。

### 未来に起きそうなこと

法令解釈、社内規程、業界の暗黙ルール。これらは、これまで「文書」と「実務担当者の頭」に分かれて存在してきました。

MCP サーバーは、その間に **沈める層** になり得ます。表面の自然言語ではなく、スキーマ・出典保持・監査ログとして、判断の根拠を企業内に留める。UI に浮いていた業務知識が、プロトコルの下に沈んで、はじめて他の判断と組み合わせられるようになります。

経営から見ると、これは「業務マニュアルを書き直す」のではなく、**「業務マニュアルを、実行可能な形でコードの下に沈める」** という変化です。

## S3. Spillovers — よい MCP は、作った人の外側に知識を漏らす

MCP の Spillovers は、ツール名・説明文・スキーマ・参照リソースとして公開されたものが、**コードを書かない人にとっても「業務の整理」として読める**、という性質です。一つのサーバーが、書いた組織の外側に対して、領域の語彙そのものを漏らしていきます。

無形投資のもうひとつの特徴は、**価値が作った企業の外側に流れ出す** ことです。基礎研究の成果は同業他社にも流れ、優れた組織運営は転職者を通じて漏れ出します。

MCP は、この性質を **設計上、最初から持っている** プロトコルです。なぜなら、MCP サーバーが公開するのは「API への接続」だけではないからです。

- **ツール名と説明文** が、その業務で重要な操作を言語化する
- **入力スキーマ** が、その領域のデータモデルを公開する
- **プロンプト雛形** が、現場で有効な聞き方の型を共有する
- **参照リソース** が、参照すべき公式情報の場所を地図として残す

ひとつの MCP サーバーは、コードを書かない人にとっても **業務の整理** として読まれ得ます。

具体例:

- デジタル庁が公開した [Jグランツ MCP](https://github.com/digital-go-jp/jgrants-mcp-server) は、補助金 API をラップしたサーバーですが、その存在自体が「行政データは MCP 化できる」という実例として広く参照されています
- [Smithery](https://smithery.ai/) や [mcp.so](https://mcp.so) のようなレジストリは、各サーバーのスキーマやアノテーションをスキャンして、業務領域ごとに地図を作ろうとしています
- ある領域の MCP サーバーを実装すると、隣接サーバーのコードを読む機会が自然に増えます。読んでいるうちに気づいた小さな改善は、issue や pull request として相手のリポジトリに戻っていきます。spillover は、一方通行ではありません

### 未来に起きそうなこと

MCP サーバーが増えれば、業務領域ごとの「最小単位の理解」が、コミュニティ全体に蓄積していきます。

会計、法務、人事、調達、研究開発、医療、教育。それぞれの領域で「最初の一つ」が公開されると、後発のサーバーはそのスキーマとプロンプトを参照しながら作られます。

つまり MCP の spillover は、**「業務の語彙」そのもの** を公共に流していく仕組みになります。これは、産業政策的にも興味深い性質です。

## S4. Synergies — MCP の本質は、複数サーバーの掛け合わせにある

MCP の Synergies は、**ホストが複数のサーバーを同時に接続でき、エージェントが複数サーバーのツール呼び出しを順に組み合わせて、新しい判断を作れる** という性質です。これは MCP のプロトコル仕様そのものに織り込まれている特性です。

ここまで見てきた S1〜S3 だけでも、MCP は無形資産としての性質を備えています。けれども、MCP の本当の意味はここにはありません。

**S4 — Synergies、掛け合わせ。**

Haskel と Westlake は、無形資産は他の無形資産と組み合わせることで、最も大きな価値を生むと指摘しました。R&D は、優れた組織と組み合わさってはじめて製品になります。ブランドは、販売チャネルと組み合わさってはじめて売上になります。

MCP は、この性質を **プロトコルそのものの形** として持っています。

ホスト（Claude / Cursor / ChatGPT 等）は、複数のサーバーを同時に接続できます。会話の中で、エージェントは複数のサーバーのツールを順番に呼び、結果を組み合わせて新しい判断を作ります。

### 具体例: 入札 × 補助金 × 会計

私たちの環境では、3 つの MCP サーバーが同時につながっています。

| サーバー | 担当 | 提供元 |
|---|---|---|
| **JP Bids MCP** | 官公需入札（KKJ） | スグクル株式会社 |
| **Jグランツ MCP** | 補助金（Jグランツ） | デジタル庁公式 OSS |
| **freee MCP** | 会計・請求書 | freee 株式会社 |

ひとつの会話の中で、こう尋ねることができます。

> 「鹿児島県の IT システム調達案件のうち、当社のキャッシュで応札可能な額のものを抽出してください。落札を見込んだ場合、当面の運転資金として活用できる補助金も合わせて教えてください。」

エージェントは、

1. 入札 MCP で該当案件を検索し、
2. 会計 MCP で現在のキャッシュ残高を取得し、
3. 補助金 MCP で受付中の補助金を取得し、
4. 三つを突き合わせて、ひとつの判断として返す

という 4 段階を、ひとつの会話の中でこなします。

これは、3 社が 3 つの SaaS として提供しても起きませんでした。**MCP という同じ形** で並べ直したからこそ、エージェントが連結できたのです。

### 未来に起きそうなこと

掛け合わせは、2 つや 3 つで止まる理由がありません。

- 公共調達 × 補助金 × 会計 × 契約 × 人材 × カレンダー
- 医療記録 × 保険 × 薬価 × 服薬カレンダー × 家族通知
- 不動産 × 都市計画 × 補助金 × 建築基準 × 工程管理

掛け算の組み合わせは、当事者の頭の中にあった「業務」の構造をそのままなぞります。誰かがホワイトボードに矢印を書いていた業務図が、ホストの中で、エージェントの呼び出しグラフとして再現されるようになります。

ここに、無形資産の最大の特性 — **使うほどに価値が増える、組み合わせるほど予想外の価値が生まれる** という現象が、ようやくソフトウェアの形で実装されます。

## 公式の思想 — なぜ MCP は最小規約のままでいられるのか

MCP が最小規約のままでいられる理由は、**3 つの登場人物（Host / Client / Server）と 7 つほどのプリミティブだけで、4S のすべてを実装可能な形に切り取っているから** です。新概念の追加よりも、既存標準（JSON-RPC 2.0）と既存設計言語の組み合わせを優先した設計判断が、無形資産的な掛け合わせの土台になっています。

[MCP 公式アーキテクチャ](https://modelcontextprotocol.io/docs/concepts/architecture) を読むと、驚くほどシンプルです。

- 3 つの登場人物（Host / Client / Server）
- JSON-RPC 2.0 という、20 年以上前からあるシンプルな通信規約
- 7 つほどのプリミティブ（Tools / Resources / Resource Templates / Prompts / Completion / Logging / Notifications）
- ステートフルな接続

新しい概念をたくさん導入していません。むしろ、既存の標準と既存の設計言語の組み合わせで成立しています。

しかし、4S の観点から見ると、この最小限の規約こそが、無形資産的な掛け合わせを可能にする土台になっています。プロトコルが小さいからこそ、領域も実装言語も越えて並べられます。

[MCP Apps](https://github.com/modelcontextprotocol/typescript-sdk) のように、ツール実行結果が会話の中に表として直接描画される拡張も、この最小規約の上に載っています。JSON でも外部リンクでもなく、**会話の続きに、業務画面が静かに置かれる**。

それは、新しい派手な機能ではありません。**同じ形で並ぶことの、当然の結果** です。

## おわりに — 掛け合わせは、どこまで続くのか

無形資産の 4S に、MCP はほとんどそのまま当てはまりました。本稿冒頭の主張、「MCP は経済学者の 4S にそのまま当てはまる」は、ここまでの 4 章で実例とともに裏付けてきたつもりです。

| S | MCP での発火の仕方 |
|---|---|
| **Scalability** | ホストを越えて、同じ形で何度でも呼ばれる |
| **Sunkenness** | 価値の本体は、ツール名の下に沈む |
| **Spillovers** | スキーマとプロンプトとして、業務の語彙が外に漏れる |
| **Synergies** | 掛け合わせるほど、予想していなかった判断が生まれる |

掛け合わせがどこまで続くのかは、誰にも分かりません。

ただ、ひとつだけ言えることがあります。**正しく掛け合わせれば、公式の思想が言うとおり、シンプルで美しいまま、強い体験を生む** ということです。

派手な変革を語る必要はありません。机のうえの一枚の紙が、いま、エージェントを通じて、補助金や会計や契約と静かに繋がりはじめています。

それは、これまでのソフトウェアが扱えなかった種類の価値です。経営者の語彙でいえば、これは「単一機能を売る SaaS の時代」から「文脈を共有するエージェントの時代」への、静かな移行が始まっているという話でもあります。

この観察が正しければ、向こう数年のソフトウェアの競争軸は、機能の比較ではなく、**どれだけ「掛け合わせ可能な形」で公開されているか** に移ります。それは、無形資産の経済学が予言してきた未来でもあります。

## よくある質問 — 経営者と開発者からの問い合わせより

### Q. MCP は、エンジニアでなくても理解しておくべきですか？
A. はい。MCP は「機能」ではなく「並べ方」の規約なので、**業務知識を持つ人ほど、その並べ方の価値が見える** という性質があります。経営者・業務責任者にとって MCP は、「自社の知識を、AI 越しに反復利用される無形資産に変える」設計の話として読めます。

### Q. 自社の業務データを MCP サーバーにする、という判断はいつすべきですか？
A. 業務に「文書化されているけれど、文書のままでは使い回せない知識」がある時です。社内規程、与信基準、契約雛形、稟議の慣習などが該当します。MCP サーバーにすることで、それらが AI 越しに反復利用される形になります。逆に、文書化すらされていない暗黙知は、まず文書化のほうが先です。

### Q. MCP と従来の API は、何が違うのですか？
A. 大きな違いは「並べ方」が標準化されている点です。従来の API は、各社が独自の形で提供してきたので、AI から使うには各 API ごとに専用のラッパーが必要でした。MCP は、ツール・リソース・プロンプトといった「並べ方の枠」を共通化するので、AI 側はどの MCP サーバーも同じ手順で扱えます。

### Q. なぜ無形資産の 4S なのですか？他の経済学フレームワークではないのは？
A. 4S は、無形資産が有形資産と「どう違うか」を最も簡潔に整理したフレームワークです（Haskel & Westlake, 2017）。MCP はソフトウェアでありながら、データセンターやサーバーといった有形資産ではなく、**プロトコル仕様という極めて無形な資産** として価値を生みます。だからこそ、4S が一番素直に当てはまります。

### Q. この記事の主張は、何かの製品宣伝ですか？
A. 結果として私が運営する [JP Bids MCP](https://mcp.bid-jp.com) を例に挙げていますが、本稿の主旨は MCP というプロトコルそのものの観察です。スグクル株式会社の事業のためにこの記事を書いたというより、自社で MCP サーバーを実装してみて気づいたことを、経済学のフレームに置いて記録した、という順序で書かれています。
